## Purpose

This document defines the editorial guidelines, spelling standards, grammatical preferences, and formatting conventions for all written content published on BridGenta. It ensures that every page is professional, readable, consistent, and matches our target B2–C1 German language standard. Every contributor—human or AI—must follow these writing rules.

---

## Target Language & Reading Level

- **Primary Language**: German (Deutsch)
- **Target Level**: B2–C1 (professional, precise, but accessible)
- **Form of Address**: Direct, respectful, and engaging (e.g., standard "Sie" or neutral professional tone for articles, direct conversational form for the portfolio interface).

---

## Grammatical & Style Rules

### 1. Prefer Active Voice (Aktiv vor Passiv)
Active sentences are clearer and more engaging than passive constructions.
- **Avoid**: *Der Code wurde durch den AI-Builder generiert.*
- **Prefer**: *Der AI-Builder generiert den Code.*

### 2. Sentence Length and Structure
Keep sentences concise. Avoid overly complex nested structures (Schachtelsätze). Aim for a maximum of 15–20 words per sentence.
- **Rule**: Limit sentences to one main idea. Break down long explanations into sequential paragraphs or bullet points.

### 3. Clear Paragraphs
No paragraph should be longer than 3–4 sentences. Short paragraphs improve scannability on desktop and mobile screens.

### 4. Bullet Points & Lists
Use bulleted lists for:
- Checklists
- Constraints or technical features
- Sequential process steps
- Key takeaways

## Technical Terminology & Formatting

### 1. English Technical Terms (Anglizismen)
In IT and web development, English terms are common. Handle them consistently:
- **First Use**: Introduce and explain technical terms (e.g., *GEO (Generative Engine Optimization — Answer-Engine-Optimierung)*).
- **Capitalization**: Capitalize English nouns integrated into German sentences (e.g., *das Commit*, *das Repository*, *das Webmaster-Service*).
- **Spelling**: Use hyphens for compound English-German terms (e.g., *AI-gestützt*, *Frontend-Entwicklung*, *Git-Workflow*).

### 2. Markdown Formatting Conventions
- **Bold (`**bold**`)**: Use to highlight key technical decisions, outcomes, or files. Do not over-use.
- **Inline Code (`` `code` ``)**: Always use for file paths, commands, code symbols, variables, and ports (e.g., [Layout.astro](file:///c:/antigravity/statichtmlpro/fdrefs/src/layouts/Layout.astro), `npm run dev`, `port 4001`).
- **Fenced Code Blocks**: Use language identifiers for code snippets to enable syntax highlighting.

### 3. Avoid Hyperbole & Marketing Jargon
BridGenta values matter-of-fact engineering quality over marketing claims.
- **Ban List**: *Revolutionär, bahnbrechend, ultimativ, perfekt, Magie, künstliche Intelligenz wird die Welt verändern.*
- **Prefer**: *Effizient, qualitätsgesichert, reproduzierbar, AI-gestützte Beschleunigung, kontrollierter Workflow.*

---

## Editorial Quality Checklist

Before completing any writing task, verify:
- [ ] Is the language professional B2–C1 German?
- [ ] Are sentences short and active?
- [ ] Are technical English nouns capitalized correctly?
- [ ] Are all code commands, files, and paths formatted as inline code?
- [ ] Is marketing hyperbole removed?
- [ ] Are compound terms correctly hyphenated (e.g., *CI/CD-Pipeline*)?
